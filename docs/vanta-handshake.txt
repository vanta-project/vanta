            Vanta Handshake and Secure Session Specification
                        draft-vanta-handshake-00

Abstract

   This document defines the Vanta authenticated handshake, session
   establishment, key derivation, and negotiation binding.

1.  Security Goals

   *  mutual peer authentication

   *  forward secrecy

   *  downgrade resistance

   *  transcript integrity

   *  transport-independent identity

2.  Cryptographic Suite

              +=========================+===================+
              | Component               | Algorithm         |
              +=========================+===================+
              | Identity signature      | Ed25519           |
              +-------------------------+-------------------+
              | Ephemeral key agreement | X25519            |
              +-------------------------+-------------------+
              | Key derivation          | HKDF-SHA-256      |
              +-------------------------+-------------------+
              | Session AEAD            | ChaCha20-Poly1305 |
              +-------------------------+-------------------+
              | Transcript hash         | SHA-256           |
              +-------------------------+-------------------+

3.  Peer Identity

   Each peer has a long-term Ed25519 identity key pair.  The public key
   bytes are the PeerID.

   Implementations MAY use pinned public keys, centralized key
   registries, or equivalent trust bootstrap mechanisms.

4.  Handshake Phases

   1.  HELLO

   2.  AUTH_INIT

   3.  AUTH_FINISH

   4.  NEGOTIATE

   5.  CAPS_CONFIRM

   Application data MUST NOT be exchanged before session activation.

5.  HELLO

   HELLO exchanges supported protocol versions, crypto suites, maximum
   frame sizes, transport profiles, and feature sets.

6.  AUTH_INIT

   AUTH_INIT conveys the initiator identity, ephemeral X25519 public
   key, freshness data, and a signature over the current handshake
   transcript and selected parameters.

7.  AUTH_FINISH

   AUTH_FINISH conveys the responder identity, responder ephemeral key,
   transcript commitment, and a signature over the handshake transcript.

8.  Key Derivation

   The shared secret is derived using X25519 over the local private
   ephemeral key and the remote public ephemeral key.

   HKDF-SHA-256 is then used to derive directional session keys, nonce
   bases, resumption material, audit binding material, and the
   SessionID.

9.  Capability and Schema Negotiation

   After peer authentication, both sides exchange schema family support,
   schema versions, capability sets, extension namespaces, and ordering
   profiles.  No capability may be used before final confirmation.

10.  Session Activation

   A session becomes active only after successful completion of HELLO,
   authenticated key exchange, encrypted negotiation, and capability
   confirmation.

11.  Active Attack Resistance

   Vanta resists active man-in-the-middle attacks by signing the
   ephemeral key exchange with long-term identity keys and binding the
   selected version, suites, and negotiated properties to the
   authenticated transcript.

12.  Resumption Foundation

   Vanta defines resumption material for reconnect optimization, but
   resumption does not imply automatic replay of unacknowledged data.
   Replay and retransmission remain explicit policy decisions.

13.  Security Considerations

   Implementations MUST verify handshake signatures, MUST reject
   transcript mismatches, and MUST NOT reuse AEAD nonces within the same
   key scope.

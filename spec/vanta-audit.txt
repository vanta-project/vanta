                    Vanta Audit Chain Specification
                          draft-vanta-audit-00

Abstract

   This document defines the Vanta audit receipt format, receipt
   chaining model, event classes, and verification rules for
   cryptographically verifiable execution tracing.

1.  Overview

   Vanta includes a protocol-native audit layer for important protocol
   and application events.  Audit receipts are intended to be compact,
   deterministic, signed, and tamper-evident.

2.  Audit Event Classes

   Audit receipts may be emitted for mutation application, duplicate
   handling, validation failures, retransmissions, and session lifecycle
   events.

3.  Receipt Format

   An audit receipt contains at least a receipt identifier, a previous
   receipt hash, session and stream identifiers, message and operation
   identifiers where applicable, sender and receiver peer identifiers,
   an event type, a disposition code, a payload hash, a timestamp, and a
   receiver signature.

4.  Payload Hashing

   Audit records MUST NOT include plaintext payload content.  Instead,
   they bind to payload bytes using a cryptographic hash.  For end-to-
   end encrypted payloads, the ciphertext SHOULD be hashed for audit
   binding.

5.  Receipt Chain Construction

   Audit receipts form an append-only hash chain.  Each receipt commits
   to the previous receipt hash.

   ReceiptHash_n = SHA256(CanonicalReceiptFields || PreviousReceiptHash)

6.  Receipt Signatures

   Audit receipts are signed by the receiver using its long-term Ed25519
   identity key.  This provides authenticity, attribution, and tamper
   evidence.

7.  Verification

   A verifier checks the receipt signature, recomputes the receipt hash,
   verifies continuity to the previous receipt hash, and validates the
   payload hash and referenced identifiers where available.

8.  Cross-Session Continuity

   Implementations MAY support cross-session audit continuity by
   referencing the terminal receipt hash of a previous session.

9.  Privacy Properties

   The audit layer intentionally stores metadata and hashes rather than
   payload contents or full application queries.  This is intended to
   balance verifiability with confidentiality.

10.  Audit Failure Handling

   If receipt generation or verification fails under a policy that
   requires audit integrity, the implementation SHOULD emit an explicit
   protocol error and MAY terminate the session.

11.  Security Considerations

   Compromise of a peer's long-term signing key can invalidate receipt
   attribution for that peer.  Implementations SHOULD protect long-term
   keys and receipt logs accordingly.

12.  Normative References
